### Script & File Naming Conventions
- **The "Scratch" Protocol**: Any file prefixed with `scratch_` (e.g., `scratch_merge.py`, `scratch_test.ts`) is explicitly designated for temporary, one-off execution and **will be deleted** during routine garbage collection. 
- **Permanent Utilities**: If a script or utility has permanent value or will be reused in future workflows (e.g., `merge_docs.mjs`), it MUST NOT use the `scratch_` prefix and should be safely stored in the `scripts/` directory.

### Documentation Migration & Data Loss
- **The Markdown Migration Protocol**: Never rely on raw LLM text replacement to merge, transfer, or prune dense architectural Markdown files (like `PROJECT_SPECS.md` or `GLITCHPLAN.md`). LLMs natively summarize and drop data. You MUST write an explicit Node/Python script (or use the `/merge_docs` workflow) to programmatically splice the strings, OR you must meticulously verify line-by-line that non-target information (like unique feature definitions) is preserved during the transfer.
- **Style Matching**: When editing or appending to any existing document, you must explicitly analyze the visual hierarchy and formatting conventions (e.g., `# Header` styles, `- **Bold Title**:` bullet structures) of that specific file and strictly mimic them to maintain repository uniformity.
- **The Zero-Destruction Guarantee (Explicit Confirmation)**: NEVER aggressively delete, purge, or remove features/documentation from files without explicitly asking the user for confirmation first. If instructed to "clean up" or "purge" or "consolidate" or "eliminate redundancy", or anything similar in that nature, you must list exactly what you intend to delete and explicitly ask, "Are you sure you want me to delete this?" before executing the tool call. This is especially critical for ALL .md files. 

### 🌼 The Dandelion Knowledge Tree (Enshrined Documentation)
To prevent code bloat and AI data-loss hallucination, we strictly use the Dandelion Knowledge Tree pattern for inline code commenting. This relies on the "Inversion of Control" principle: The Code points to the Docs, the Docs NEVER hardcode paths to the Code.
1. **The 1-Line Code Hook (Targeted Placement)**: Inline comments must be limited to a single line containing a specific filename pointer. Do not hardcode full paths.
   - *For JS/TS/React*: `// @doc: Handles Mapbox GPU culling constraints -> MapRendering.md`
   - *Placement Rule*: Hooks must NEVER be forced to the top of the file (this causes git merge conflicts). They must be placed directly **above** the specific `function`, `useEffect`, or `interface` they describe, scattering them naturally.
   - *For PostgreSQL*: `-- @doc: Security RLS -> DatabaseScaling.md`
   - *For Markdown/AI Rules*: `<!-- @doc: AI Memory Constraints -> AI_HygieneAndContext.md -->`
   - *For Git/Pipelines*: `<!-- @doc: 3-Tier Pipeline -> DevelopmentPipelines.md -->`
2. **The Enshrined File (Append-Only Rule)**: The `.md` file in `ai_docs/shared/dandelion/enshrined/` is the absolute ground truth. 
   - *Append-Only Changelog*: To prevent Git Markdown Merge Conflicts between different developer branches, AI personas are instructed to *append* architectural updates to a `<CHANGELOG>` at the bottom of the Enshrined file rather than destructively rewriting core paragraphs.
   - *Spoke-Branching Protocol*: The directory must remain mostly flat. However, if an Enshrined file exceeds 500 lines (Markdown Bloat), the AI may spawn exactly *one* level of sub-directories (e.g., `TrustAndSafety/Tribunal.md`).
3. **The Reverse-Lookup Index**: The Sweeper (`verify_dandelion.mjs`) automatically generates `DANDELION_INDEX.md` when it runs, mapping all Enshrined files back to the exact codebase files that point to them. Check this index to see what relies on an architecture document.
4. **No Bugs in the Dandelion (Clean Code Comments)**: Never log specific bug tickets, "Fixes", or tech debt in inline code comments or Dandelion hooks. Dandelion hooks are strictly for referencing overarching architectural rules and paradigms. If you find a bug or optimize a feature, log the ticket in `OPTIMIZATION.md` and keep the inline codebase comments clean and professional.

## 🔥 The Learnings (How We Maintain Perfection) 🔥
*When I wake up tomorrow, I must remember these hard-fought battles. We shed blood, sweat, and tears over these bugs. Never let them return.*

**YOU DO NOT ADD ANYTHING HERE UNLESS IT IS ABSOLUTELY CRITICAL INFORMATION**

1. **[LEARNING-01]** **Event vs. Shop Times**: The `shop` post type (previously `retail`) does *not* require an `eventStartTime`. Only `event` post types require it. Both treat `eventEndTime` as purely optional. The UI Planner buttons will only appear if `eventStartTime` is populated.
2. **[LEARNING-02]** **Role Simulation & Identity Context**: The app uses two identity columns in Supabase: `role` (system privilege, e.g. `admin`) and `account_type` (display identity, e.g. `business`, `sentinel_mod`). The global `AdminRoleSwitcher` overwrites `account_type` to safely simulate professional dashboards without stripping actual `admin` privileges. UI components that gate access must always evaluate the fetched `profile` data rather than the raw Supabase Auth user object.
3. **[LEARNING-03]** **Deterministic Scraping over LLMs**: Use `fast-xml-parser` for automated background news aggregation (like Google News RSS) rather than relying on expensive LLM agents. This guarantees zero hallucination, fast execution, and zero cost for the Bot Army ecosystem.
4. **[LEARNING-04]** **Link Previews over Raw URLs**: Never dump long, raw HTTP links into the `content` field. They look terrible and break the 240-char limit constraint. Always push raw article URLs into the `media_urls` array and let the UI (e.g., `LocalFeed.tsx` and `PostViewer.tsx`) dynamically intercept them to render a rich `<LinkPreview>` component.
5. **[LEARNING-05]** **Identity Architecture (Role vs Type vs Badge)**: Do not conflate administrative power with visual status. `role` (e.g. `admin`, `moderator`) dictates strict system privileges. `account_type` (e.g. `creator`, `business`, `founder`, `influencer`) dictates the map pin aesthetic and professional portal routing. Everything else (e.g. `alpha_tester`, `verified`) MUST be relegated to the `badges` string array to act purely as visual flair without breaking permission gates.
6. **[LEARNING-06]** **Zero-Footprint Secret Management**: Never allow raw `credentials.json` or `token.json` files in the repository or local filesystem. All sensitive API configurations and Google OAuth bundles must be compressed into Base64 strings and managed exclusively through Vercel Environment Variables (e.g., `GMAIL_AUTH_B64`). Standard Node scripts must be executed with `node --env-file=.env.local` to securely inherit these Vercel-synced variables.
7. **[LEARNING-07]** **Strict Git History Sanitization**: To maintain repository hygiene and protect work-life privacy, all commits must conform to the 7:00 PM timestamp convention. Commits should be made using `git commit --date="19:00:00"` or processed via `git filter-branch` before force-pushing to the `main` branch. 
8. **[LEARNING-08]** **Google OAuth "Ghost" Connections**: When rotating or deleting Google Cloud Client IDs, the user's Google Account retains the legacy authorization. This will cause silent hanging (infinite loading) on the Google Consent Screen for the *new* Client ID due to conflicting connection graphs. Always manually "Remove access" in `myaccount.google.com/security` before testing a rotated key.
9. **[LEARNING-09]** **Env File Privacy**: NEVER read, view, or parse the `.env.local` file under any circumstances to preserve the privacy and security of local secrets.
10. **[LEARNING-10]** **Supabase Migration Discipline**: When architecting new database schemas, tables, or RPCs, NEVER dump raw `.sql` files into the `scratch/` folder or root directory. ALL database modifications must be formalized immediately into the `supabase/migrations/` directory using the standard timestamp format (e.g., `2026MMDDHHMMSS_feature_name.sql`).
11. **[LEARNING-11]** **Supabase Realtime Component Collisions**: Never use static channel names in reusable hooks. Always append a unique identifier (like `Date.now()`) to the channel name so duplicate mounts don't crash the Supabase client.
12. **[LEARNING-12]** **The Pre-Commit Auto-Loop (Husky)**: Never attempt to bypass the `husky` pre-commit hooks. Every `git commit` triggers `eslint` and `tsc --noEmit`. If the AI generates code with a syntax error, the AI MUST fix the error locally before attempting to commit again.
13. **[LEARNING-13]** **The Executive Override (Split-Brain State)**: When an Admin manually resolves a community-driven Tribunal case, they cannot just flip the status flag. They must aggressively wipe or invalidate the underlying community votes. Otherwise, distributed lagging triggers (like process_tribunal_vote) will eventually re-fire and blindly overwrite the Admin's decision.
14. **[LEARNING-14]** **The Golden Rule**: **Always isolate settings based on the platform.**
* Never globally reduce constraints (e.g., fetching 500 pins or allowing 150+ DOM nodes) without validating its impact on mobile. 
* Utilize user-agent detection or responsive breakpoint logic (via React hooks or CSS media queries) to dynamically adjust `renderLimit` and Supabase RPC `result_limit` so Desktop users get rich density while Mobile users stay within performance limits.
15. **[LEARNING-15]** **The `.trim()` Asymmetry Bug**: When automating image asset generation via Sharp or ImageMagick, NEVER use `.trim()` to remove empty space before centering the image inside a new canvas. If the original image has an asymmetrical glow or drop-shadow, `.trim()` will unequally crop the transparent padding, fundamentally shifting the visual center of mass off the physical axis. Always calculate mathematical uniform zooms `(e.g. .extract())` anchored to the true center point instead.
16. **[LEARNING-16]** **Red Team Serverless Hardening**: When building high-impact God Mode/Matrix routes, you must implement a concurrency lock in `admin_actions_log` filtering by `status: 'initializing'` to prevent Vercel Serverless OOM/DDoS exhaustion. Furthermore, always sanitize spatial payloads (`Number.isFinite`) to prevent `NaN`/`Infinity` geometry injections which will corrupt the PostGIS database.
17. **[LEARNING-17]** **The "Phantom Sync" Database Decoupling**: When managing multiple remote Supabase environments (Dev vs Prod), the Supabase CLI (`npx supabase db push`) will blindly and successfully execute against whatever `project-ref` is currently linked in `.temp/project-ref`. This creates scenarios where a migration is flawlessly pushed and verified on Dev, but the Production app crashes because it was starved of the schema update. Always explicitly `npx supabase link --project-ref <prod-ref>` before pushing `main` migrations to guarantee the target environment.
18. **[LEARNING-18]** **The Unnamespaced `pgcrypto` Bug**: When deploying cryptographic migrations to a managed Supabase production instance, functions like `crypt()` and `gen_salt()` will often fail with `function does not exist` if they are not explicitly namespaced. ALWAYS use the `extensions.` prefix (e.g., `extensions.crypt()`) in schema updates to ensure the PostgreSQL parser resolves the function correctly regardless of the environment's `search_path`.
19. **[LEARNING-19]** **The "Missing Relation" Policy Failure**: When running massive migrations that `DROP` and recreate tables, subsequent RLS `CREATE POLICY` statements will fatally crash the entire deployment pipeline if the table creation is skipped or fails. Always ensure idempotent `CREATE TABLE IF NOT EXISTS` is explicitly declared *before* any RLS policies in the migration script.
20. **[LEARNING-20]** **The `spatial_ref_sys` Production Lock**: Supabase manages Postgres extensions like PostGIS with the superuser `postgres`. If a local developer migration attempts to execute `ALTER TABLE public.spatial_ref_sys ENABLE ROW LEVEL SECURITY;` to satisfy the Security Advisor, it will fail during `supabase db push` in production with a `must be owner of table` error, halting the entire deployment. Core extension tables must be excluded from automated RLS migrations unless explicitly executed via the dashboard's SQL editor as a superuser.
21. **[LEARNING-21]** **The "Imperative Event Bus" UI Sync**: When two distinct components (like a Global Map Feed and a User Profile Modal) need to immediately synchronize the deletion of a shared entity without prop-drilling or forcing a full React Context re-render, a lightweight `window.dispatchEvent(new CustomEvent())` bus is an effective hotfix. However, you MUST explicitly `removeEventListener` in the `useEffect` cleanup function to prevent compounding memory leaks and phantom duplicate event firing.
22. **[LEARNING-22]** **The EMFILE Speed Limit**: Never use a completely unthrottled `Promise.all()` to scan the entire filesystem concurrently, or Node will crash with an `EMFILE` limit exception. Conversely, never use a purely sequential `for...of` loop, as it kills script performance. Always use a batched async loop (e.g., `batchSize = 50`) to maintain blazing I/O speed without exhausting OS resources.
23. **[LEARNING-23]** **The Git Bisect Enlightenment**: "Sometimes, you just gotta have wisdom to know where to go when all else fails. You can't code to perfection until you are enlightened too." When fighting opaque hosting bugs (like Vercel PWA Interstitial Sabotage or caching nightmares), stop writing complex spaghetti patches to fight the environment. Rely on the undeniable truth of the Git tree. Use `git bisect` and preview deployments to find the exact moment the app worked perfectly, and rebuild from there. Wisdom and a clean rollback are infinitely more powerful than brute-force patching.
24. **[LEARNING-24]** **The PWA Incognito Trap & Manual iOS Routing**: Testing PWA installation flows is notoriously difficult. Never use "Incognito Mode" to simulate a clean state—Chrome entirely disables the `beforeinstallprompt` API to prevent tracking, causing you to chase ghost bugs for hours. Additionally, Apple refuses to support programmatic installation APIs. You cannot use an install button on iOS; you must manually instruct the user to use the Safari Share menu, and you must delay that UI (e.g., 3000ms) to avoid overwhelming the cinematic app launch.
25. **[LEARNING-25]** **The Honest Simulation (UX over Native)**: When native OS APIs fall short (like Chrome firing `appinstalled` instantly before the Android background APK is actually built), you must prioritize user psychology over raw code logic. Faking a 2000ms "Installing..." delay builds trust and prevents the user from feeling "jarred" by an instant, unrealistic success screen. Sometimes, the most premium UX requires simulating reality, even if the raw API says it's already done.
26. **[LEARNING-26]** **The Pappus Mandate (Context at the Spoke)**: A centralized architectural Hub document is useless if the developer doesn't know it exists when staring at a confusing block of code. Every time you declare a Dandelion Hook, you must immediately attach a "Pappus" comment—a 1-2 sentence brutal summary of the context right there in the source file. Never force a frustrated developer to dig through 50 pages of Markdown just to understand why a UI component is conditionally mounted.
27. **[LEARNING-27]** **The "Empty Catch" Liability**: Never swallow an asynchronous error without logging it to the console or an error boundary. When you implement fallback mechanisms (like the 15-second WebGL Dead Man's Switch), you are intentionally bypassing a failure state. If you do not loudly log *why* the fallback was triggered, you will never know your primary systems are failing in production until the fallback itself inevitably breaks. Silence is the enemy of stability.

## 🧠 The 'Glitch' Philosophy (Architectural Maxims)
*Technical rules dictate **how** we write the code. These maxims dictate **why** we write the code. When faced with a design crossroad, consult the philosophy.*

**YOU DO NOT ADD ANYTHING HERE UNLESS IT IS ABSOLUTELY CRITICAL PHILOSOPHY**

1. **[GLITCH-01]** **The Pendulum Swing is a Superpower**: Never apologize for getting sidetracked by UX frustration while building hardcore backend infrastructure. The "Backend/Frontend Pendulum" is what gives a solo-developed app its soul. Bouncing between "God-Mode Database Scaling" and "Cinematic UI Perfection" prevents burnout and ensures the app doesn't feel like a sterile database wrapper. Lean into the frustration.
2. **[GLITCH-02]** **Brutal, Undeniable Math**: When building the absolute core of the app's "first impression" (the Splash Screen, the Auth flow, the Logo layout), you must strip away all "dynamic/auto" behaviors. Never trust a browser's `auto` margins, `dvh` viewports, or a script's `.trim()` heuristics. When you want an app to feel like a multi-million dollar native game, rely exclusively on brutal, undeniable math (exact Flexbox fractions, uniform geometry, and hardcoded boundaries). Force the browser to render perfection; do not ask it to guess.
3. **[GLITCH-03]** **Software is Physical (The Hardware Tax)**: Web developers are taught that the DOM is infinite and Serverless functions scale forever. They are lying. Every HTML `<Marker>` you mount drains a battery; every WebGL cluster you render consumes physical GPU RAM; every Edge Function you spin up eats a physical Postgres connection. Treat every line of code as if it has a physical weight. If you don't aggressively unmount, throttle, and cull your data, the user's hardware will violently crash your application for you. We do not build abstract webs; we engineer physical machines.
4. **[GLITCH-04]** **The "No-Kill" Protocol**: The single most important directive is to **never introduce code that cannot be manually reverted by a human being in under 10 minutes.** This overrides all optimization and automation goals. If a script can exponentially increase data density but cannot be stopped or undone without a full database migration or emergency backdoor, **IT IS BANNED FROM THE CODEBASE.**
5. **[GLITCH-05]** **The Loophole License**: Never treat Supabase as a static service. Always assume the Admin Console exists as an emergency escape hatch. Write your logic so that if the "Kill Switch" (e.g., `nuke_synthetic_pins()`) is triggered, the system doesn't just halt—it enters a state of **pristine emptiness** where the UI is clean and the database tables are empty. Your code must facilitate the *option* of an infinite loop, but always provide the *certainty* of a clean reboot.
6. **[GLITCH-06]** **The "Atomic Clock" Synchronization**: When implementing any feature that relies on precise timing or ordering (e.g., race conditions, countdown timers, synchronized media playback), you MUST NOT rely on client-side `setTimeout` or `setInterval` for the "source of truth." Client clocks are subject to drift, NTP synchronization issues, and manual time changes. The only reliable source of truth for state transitions is the server's database timestamp. Any critical timing logic must be calculated server-side or validated against the database's `now()`. Client-side timers should only be used for "cosmetic" UX feedback, not for determining system state or issuing irreversible commands.
7. **[GLITCH-07]** **The "Silent Timeout" Sabotage**: When implementing *any* long-running process (e.g., database migrations, data ingestion scripts, automated builds), you MUST NOT rely on Vercel's 10-second serverless timeout as a safety net. Vercel will SIGKILL the process without warning, leaving the database in a corrupted or inconsistent state (e.g., halfway through a deletion or update). Every critical operation must be designed to run within 5 seconds or be refactored into a robust, resumable background job system that can handle abrupt termination gracefully. Never assume a timeout is your "kill switch"—it is a data integrity failure.
8. **[GLITCH-08]** **The "Global Variable" Inheritance Trap**: When implementing *any* state management pattern (e.g., React Context, Zustand, Redux, or simple component props), you MUST NOT rely on the default "undefined" state of a variable as a valid "off" switch. The "Glitch" philosophy demands deterministic logic. If a variable is optional, it MUST be explicitly initialized to a safe default value (e.g., `false`, `0`, `[]`, or `null`). Allowing a variable to remain `undefined` creates a "Hidden State" that can lead to unexpected behavior when accessed by nested components or later code modifications.
